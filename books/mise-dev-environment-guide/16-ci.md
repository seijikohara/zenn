title: "CI への組み込み"
---

本章では、GitHub Actions で mise を使い、ローカルと同じツール構成で CI（Continuous Integration、継続的インテグレーション）を回す方法を扱います。読了後には、次の操作ができるようになります。

- `jdx/mise-action` でワークフローに mise を組み込む。
- `mise.toml` のツールを導入する。
- 第 14 章で定義したタスクを CI から実行する。

本章は、開発環境を標準化したいリーダー・運用者を主な対象とします。第 3 章で mise を導入・有効化し、第 5 章でバージョン操作を理解した状態を前提とします。本章が参照する前提事項と対応する章は次のとおりです。

| 前提事項 | 扱う章 |
| --- | --- |
| `mise.toml` の構文 | 第 4 章 |
| タスクを扱う `[tasks]` セクション | 第 14 章 |
| ロックファイルと `mise trust` | 第 15 章 |

ローカルは macOS、CI は GitHub Actions の Linux ランナーを想定します。GitHub Actions のワークフロー構文そのものには深入りせず、mise の組み込みに集中します。

本章のコマンド出力やバージョン番号は、執筆時点（2026 年 6 月、mise 2026.6.10[^mise-version]）の値です。実行する時期や設定によって、表示される内容は変わります。

## mise-action の導入

GitHub Actions で mise を使う基本は、公式アクション `jdx/mise-action` をワークフローのステップに置くことです[^mise-action]。アクションは、続くステップで使えるよう次の処理をまとめて実行します[^mise-action]。

1. ランナーに mise をインストールする。
2. リポジトリの `mise.toml` を読み取って `[tools]` のツールを導入する。
3. 導入したツールを続くステップで使えるよう PATH を通す。

ローカルでの導入（第 3 章）やシェルの有効化を、CI 向けにアクションが肩代わりします。

### 基本のステップ

アクションは `uses` キーで指定します。次の例は、リポジトリをチェックアウトしてから `jdx/mise-action` を実行する最小構成です。アクションのバージョンは、執筆時点の最新メジャーバージョン `v4`[^mise-action-releases] を指定します。

```yaml:.github/workflows/ci.yml
name: CI

on: push

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: jdx/mise-action@v4
      - run: node --version
```

`actions/checkout` は、リポジトリの内容をランナーへ取得する GitHub 公式アクションです[^checkout]。`mise.toml` を含むリポジトリの内容を取得するため、`jdx/mise-action` より前に置きます。`jdx/mise-action` のステップが完了すると、`mise.toml` の `[tools]` に宣言したツールが PATH 上で有効になります。後続の `run` ステップでは、`mise exec` や `mise run` で包まずに、ツールのコマンドを直接呼び出せます。前掲の例では、導入した Node.js の `node --version` をそのまま実行します。

:::message
`jdx/mise-action` は、内部で `mise install` を実行してツールを導入します。第 5 章で扱った `mise install` を、ワークフローのステップとして実行する形です。アクションを置いたあとのステップでは、`mise install` を別途書く必要はありません。
:::

### アクションのバージョンを固定する

`uses` で指定するアクションのバージョンは、`@v4` のようにメジャーバージョンのタグで固定します[^mise-action]。`jdx/mise-action` は `v1`・`v2`・`v3`・`v4` のメジャータグを提供し、執筆時点の最新は `v4`（最新リリースは v4.1.0、2026 年 6 月 4 日公開）です[^mise-action-releases]。固定するタグには 2 つの選択肢があります。

- `@v4` のようなメジャータグ: 同じメジャーバージョンの範囲で最新のパッチへ追従します。
- `@v4.1.0` のようなパッチまで含むタグ: より厳密に固定します。

:::message
アクションのバージョン `@v4` は、`jdx/mise-action` 自体のバージョンであり、mise 本体のバージョンとは異なります。mise 本体のバージョンは、後述する `version` オプションで指定します。両者を混同しないよう注意します。
:::

## ツールのインストールとキャッシュ

`jdx/mise-action` は、`with` キーで動作を調整します。本節では、ツールのインストールとキャッシュに関する主要オプションを扱います。

### with の主要オプション

`with` で指定する主なオプションを次の表にまとめます[^mise-action]。値の既定は、アクションの `action.yml` に定義されたものです。

| オプション | 役割 | 既定値 |
| --- | --- | --- |
| `version` | インストールする mise 本体のバージョンを固定する | 未指定（最新リリース） |
| `install` | ツールを `mise install` で導入するか | `true` |
| `install_args` | `mise install` に渡す追加の引数 | 未指定 |
| `cache` | ツールのキャッシュを使うか | `true` |
| `cache_key_prefix` | キャッシュキーの接頭辞 | `mise-v1` |
| `experimental` | mise の試験的機能を有効にするか | `false` |
| `working_directory` | アクションを実行するディレクトリ | 未指定（リポジトリのルート） |

`install` と `cache` は既定で `true` です。`jdx/mise-action` を置くだけで、ツールの導入とキャッシュの両方が有効になります。

### mise 本体のバージョンを固定する

`version` オプションは、ランナーにインストールする mise 本体のバージョンを固定します[^mise-action]。未指定の場合、アクションは mise の最新リリースを取得します。CI の結果を mise 本体のバージョンに依存させたくない場合は、`version` を明示します。次の例は、mise 2026.6.10 を固定します。

```yaml:.github/workflows/ci.yml
      - uses: jdx/mise-action@v4
        with:
          version: 2026.6.10
```

ローカルで使う mise 本体のバージョンと CI のバージョンをそろえると、ローカルと CI で mise の挙動が一致します。チームで mise 本体のバージョンを管理する場合は、`mise.toml` のトップレベルに `min_version` を宣言する方法もあります[^min-version]。

### キャッシュの仕組み

`jdx/mise-action` は、`cache` オプションが `true` のとき、導入したツールを GitHub Actions のキャッシュに保存します[^mise-action]。次回以降のワークフロー実行では、キャッシュを復元してツールの再ダウンロードを省きます。キャッシュキーは、次の要素などを組み合わせて構築します[^mise-action]。

- mise の全設定ファイルのハッシュ
- ランナーのプラットフォーム
- mise 本体のバージョン
- `cache_key_prefix`

`mise.toml` を変更すると設定ファイルのハッシュが変わり、キャッシュキーも変わるため、変更後はツールを導入し直します。

:::message
キャッシュキーの接頭辞は `cache_key_prefix` オプションで変更できます。既定値は `mise-v1` です。キャッシュを意図的に破棄して導入をやり直したい場合は、接頭辞を別の値に変更します。接頭辞が変わるとキャッシュキー全体が変わり、過去のキャッシュは参照されません。
:::

## mise run でタスクを実行する

`jdx/mise-action` でツールを導入したら、第 14 章で定義したタスクを CI のステップから実行します。ローカルで使うタスクをそのまま CI で動かすことで、ローカルと CI の手順を一致させます。

### CI のステップで mise run を呼び出す

タスクは、`run` ステップで `mise run <タスク名>` を実行します[^cli-run]。次の例は、第 14 章で定義した `test` タスクを CI で実行します。

```yaml:.github/workflows/ci.yml
      - uses: jdx/mise-action@v4
      - run: mise run test
```

CI で実行するタスクは、ローカルと同じ `mise.toml` の `[tasks]` に定義したものです。ローカルで `mise run test` を実行する開発者と、CI のワークフローが、同じタスク定義を共有します。タスクが `depends` で依存タスクを宣言している場合、CI でも依存をたどって実行します。第 14 章で `test` が `lint` に依存する構成を定義した場合、CI の `mise run test` は `lint` を先に実行します。

次の例は、`lint` と `test` を別々のステップとして実行する構成です。ステップを分けると、GitHub Actions のログでどちらが失敗したかを把握しやすくなります。

```yaml:.github/workflows/ci.yml
      - uses: jdx/mise-action@v4
      - run: mise run lint
      - run: mise run test
```

:::message
ドキュメントやワークフローでは、`mise <タスク名>` の短縮形ではなく `mise run <タスク名>` を使います。第 14 章で扱ったとおり、mise が将来タスク名と同じサブコマンドを追加した場合、短縮形ではサブコマンドが優先されます。CI のワークフローでは `mise run` を明示して、タスクを確実に実行します。
:::

## ローカルと CI の再現性

ローカルと CI で同じバージョンのツールを使うには、第 15 章で扱ったロックファイル `mise.lock` を CI でも利用します。`mise.lock` に固定したバージョンとチェックサムで導入することで、CI のツール構成をローカルと一致させます。

### ロックファイルを CI で使う

第 15 章で扱ったとおり、`mise.lock` は解決済みのバージョンとプラットフォーム別のチェックサムを記録するロックファイルです。`mise.lock` をリポジトリにコミットしておくと、`jdx/mise-action` はチェックアウトしたリポジトリの `mise.lock` を読み取り、固定されたバージョンでツールを導入します。ローカルで `mise.lock` を生成してコミットし、CI で同じ `mise.lock` を使うことで、ローカルと CI のバージョンがそろいます。

`jdx/mise-action` は、v4.1.0 以降、リポジトリに `mise.lock` がある場合に `mise install --locked` を自動で実行します[^mise-action-releases]。`--locked` は、ロックファイルに記録済みの URL がないツールのインストールを失敗させるオプションです（第 15 章）[^settings-locked]。`mise.lock` をコミットしておけば、`with` に `install_args` を追加しなくても、アクションが `--locked` を付けて厳密に再現します。

### MISE_LOCKED で明示的に固定する

`mise.lock` の自動検出に頼らず、`locked` モードを明示的に有効にする方法もあります。`locked` は、`MISE_LOCKED` 環境変数でも指定できます[^settings-locked]。ワークフローのジョブに環境変数を設定すると、ジョブ内の全ステップで `locked` が有効になります。次の例は、ジョブ全体で `MISE_LOCKED` を設定します。

```yaml:.github/workflows/ci.yml
jobs:
  test:
    runs-on: ubuntu-latest
    env:
      MISE_LOCKED: "1"
    steps:
      - uses: actions/checkout@v6
      - uses: jdx/mise-action@v4
      - run: mise run test
```

`locked` を使うには、`lockfile` 設定を有効にしておく必要があります（第 15 章）[^settings-locked]。`mise.lock` をコミットして共有し、CI で `MISE_LOCKED` を設定すると、ロックファイルに固定したバージョンとチェックサムだけでツールを導入します。GitHub への API 呼び出しでバージョンを解決する処理を避けるため、CI の実行結果が安定します。

:::message
`jdx/mise-action` v4.1.0 以降の自動 `--locked` と、`MISE_LOCKED` 環境変数は、いずれも `locked` モードを有効にします。`mise.lock` をコミットしている場合、アクションが自動で `--locked` を付けるため、多くの場合は `MISE_LOCKED` の明示は不要です。アクションのバージョンに関わらず確実に固定したい場合や、自動検出の対象外の構成では、`MISE_LOCKED` を明示します。
:::

## ワークフローの実例

本章で扱った内容をまとめて、`.github/workflows/ci.yml` の全体を示します。`main` への push とプルリクエストで、リポジトリをチェックアウトし、`jdx/mise-action` で mise とツールを導入し、`mise run test` でテストを実行します。

```yaml:.github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    env:
      MISE_LOCKED: "1"          # mise.lock に固定したバージョンで導入する
    steps:
      - uses: actions/checkout@v6
      - uses: jdx/mise-action@v4
        with:
          version: 2026.6.10    # mise 本体のバージョンを固定する
      - run: mise run lint      # 第 14 章で定義した lint タスク
      - run: mise run test      # 第 14 章で定義した test タスク
```

ワークフローが参照する `mise.toml` は、第 14 章・第 15 章で扱ったものです。次の例は、Node.js のバージョンを固定し、ロックファイルを有効にし、`lint` と `test` のタスクを定義した `mise.toml` です。

```toml:mise.toml
[settings]
lockfile = true            # mise.lock でバージョンを固定する

[tools]
node = "22"                # 解決されるバージョンは mise.lock で固定

[tasks.lint]
description = "ソースコードを静的解析する"
run = "eslint ."

[tasks.test]
description = "テストを実行する"
run = "vitest run"
```

`mise.toml` と `mise.lock` をリポジトリにコミットすると、ローカルの開発者と CI が同じ定義を共有します。開発者はローカルで `mise run test` を実行し、CI のワークフローは同じ `mise run test` を実行します。`MISE_LOCKED` により、CI は `mise.lock` に固定したバージョンとチェックサムでツールを導入するため、ローカルと CI でツール構成が一致します。mise の CI への組み込みの詳細は、mise 公式ドキュメントの継続的インテグレーションのページで扱います[^mise-ci]。

## 本章のまとめ

- GitHub Actions で mise を使う基本は、`jdx/mise-action` をワークフローのステップに置くことです。アクションは mise をインストールし、`mise.toml` の `[tools]` のツールを導入し、PATH を通します。バージョンは `@v4`（執筆時点の最新メジャーバージョン）で固定します。
- `with` の主要オプションは、mise 本体のバージョンを固定する `version`、ツールを導入する `install`（既定 `true`）、キャッシュを使う `cache`（既定 `true`）です。アクションを置くだけで、ツールの導入とキャッシュが有効になります。
- キャッシュは、導入したツールを GitHub Actions のキャッシュに保存し、再ダウンロードを省きます。キャッシュキーは設定ファイルのハッシュなどから構築し、`mise.toml` を変更すると更新されます。
- 第 14 章で定義したタスクは、CI のステップで `mise run <タスク名>` を実行します。ローカルと CI が同じタスク定義を共有します。
- ローカルと CI の再現性は、`mise.lock`（第 15 章）で担保します。`jdx/mise-action` は v4.1.0 以降、`mise.lock` があると `mise install --locked` を自動で実行します。`MISE_LOCKED` 環境変数で `locked` を明示する方法もあります。

[^mise-action-releases]: `jdx/mise-action` のリリース（執筆時点の最新メジャーバージョン `v4`・最新リリース v4.1.0・`mise.lock` がある場合に `mise install --locked` を自動実行する点）。GitHub「jdx/mise-action releases」<https://github.com/jdx/mise-action/releases>
