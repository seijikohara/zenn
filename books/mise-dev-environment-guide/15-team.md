---
title: "チーム共有と再現性"
---

本章では、`mise.toml` をチームで共有し、新たに参加した開発者が `git clone` から短時間で同じツール構成に到達する運用を扱います。読了後には、何をコミットして何を除外するか、ロックファイルでバージョンを固定する方法、`mise trust` で設定ファイルを信頼する手順を理解し、再現性のあるチーム開発環境を構築できるようになります。

本章は、開発環境を標準化したいリーダー・運用者を主な対象とします。第 3 章で mise を導入・有効化し、第 5 章でバージョン操作を理解した状態を前提とします。mise 自体のインストールは第 3 章、`mise.toml` の構文は第 4 章、`mise install` / `mise use` / `mise ls` などの操作は第 5 章で扱いました。環境変数を扱う `[env]` セクションは第 13 章、タスクを扱う `[tasks]` セクションは第 14 章で扱いました。CI での環境の再現は第 16 章で、asdf などからの移行は第 17 章で扱います。本章はローカルでのチーム運用に集中します。

本章のコマンド出力やバージョン番号は、執筆時点（2026 年 6 月、mise 2026.6.10[^mise-version]）の値です。実行する時期や設定によって、表示される内容は変わります。

## 設定ファイルの共有

チームで開発環境を統一する基点は、`mise.toml` をバージョン管理システムにコミットして共有することです。`mise.toml` には、プロジェクトで使うツールのバージョン（`[tools]`）、環境変数（`[env]`）、タスク（`[tasks]`）を宣言します。1 つのファイルに開発環境の定義を集約し、リポジトリに含めることで、チームの全員が同じ定義を取得できます。

### コミットするファイル

次のファイルは、リポジトリにコミットして共有します。

| ファイル | 役割 |
| --- | --- |
| `mise.toml` | プロジェクトの標準の設定ファイル。`[tools]`・`[env]`・`[tasks]` を宣言する |
| `mise.lock` | ツールのバージョンとチェックサムを固定するロックファイル。後述する |
| `.mise/tasks/` などのファイルタスク | スクリプトとして定義したタスク（第 14 章） |

`mise.toml` は、第 4 章で扱ったとおりプロジェクトのルートに置きます。`mise.toml` をコミットすると、チームの全員が同じバージョンのツールと同じ環境変数・タスクの定義を共有できます。`mise.lock` を併せてコミットすると、解決されるバージョンまで全員でそろえられます。`mise.lock` の生成と役割は後述します。

### コミットしないファイル

次のファイルは、`.gitignore` に追加してコミット対象から除外します。

| ファイル | 除外する理由 |
| --- | --- |
| `mise.local.toml` | 個人用の上書き設定（第 4 章）。開発者ごとに内容が異なる |
| `.env` | 機密情報を含む環境変数ファイル（第 13 章）。値をリポジトリに残さない |
| `mise.local.lock` | `mise.local.toml` に対応するロックファイル。個人用の設定に紐づく |

`mise.local.toml` は、第 4 章で扱った個人用の上書き設定です。開発者が手元だけ別バージョンを使う、個人の作業環境に固有のパスを足す、といった用途に使います。チーム共有の `mise.toml` を変更せずに各自が上書きするためのファイルであり、内容は開発者ごとに異なります。コミットするとチームの設定と競合するため、除外します。

機密情報を含む `.env` は、第 13 章で扱ったとおりコミットしません。`.env` の値をリポジトリの履歴に残さず、各開発者が手元で用意します。チームには値を伏せた `.env.example` を共有し、各自が値を埋めて `.env` を作る運用がとれます。`.env.example` はコミットして共有します。

次の例は、チーム共有のプロジェクトで除外するファイルを `.gitignore` に並べたものです。

```text:.gitignore
mise.local.toml
mise.local.lock
.env
.env.local
```

:::message
mise の暗号化機能（sops・age、第 13 章）を使うと、暗号化した `.env` をコミットして共有できます。暗号化したファイルは復号鍵を持つ環境でだけ値を展開するため、平文の機密情報をリポジトリの履歴に残しません。暗号化機能は試験的機能（experimental）です。設定方法は mise 公式ドキュメントの secrets ページで扱います[^secrets]。
:::

## mise install で環境をそろえる

`mise.toml` を共有すると、新たに参加した開発者は `mise install` でプロジェクトのツールを一括で導入できます。`mise install` は、引数を指定せずに実行すると、現在のディレクトリで読み込まれる `mise.toml` の `[tools]` に宣言された全ツールをインストールします[^cli-install]。第 5 章では個別のツールを対象に `mise install` を扱いましたが、引数なしの `mise install` は設定ファイル全体を対象とします。

リポジトリをクローンした開発者は、プロジェクトのディレクトリで `mise install` を一度実行すると、`[tools]` のツールがすべて手元にそろいます。ツールごとに導入コマンドを実行する必要はありません。

```shell
$ mise install
mise node@22.20.0 ✓ installed
mise python@3.13.7 ✓ installed
```

`mise install` は、ツールを並列にインストールします[^cli-install]。インストール先は `~/.local/share/mise/installs/<ツール>/<バージョン>` です[^cli-install]。インストールだけでは、ツールは PATH 上で有効になりません。第 3 章で扱った `mise activate` を有効にしておくと、プロジェクトのディレクトリへ入ったときに mise がインストール済みのツールを PATH 上で解決します。

:::message
`mise install` はツールを導入するだけで、`mise.toml` は書き換えません。`mise.toml` にツールを追加する操作は、第 5 章で扱った `mise use` です。チーム共有のプロジェクトでは、`mise.toml` を編集してツールを追加し、コミットして共有します。各開発者は `mise install` で追加されたツールを取得します。
:::

## ロックファイルによるバージョンの固定

`mise.toml` の `[tools]` は、`node = "22"` のような緩いバージョン指定を書けます（第 4 章）。緩い指定は、解決されるパッチバージョンが解決の時期によって変わります。あるメンバーが 22.20.0 を、別のメンバーが後日 22.21.0 を導入する、といった差が生じます。完全な再現性を求める場合は、ロックファイルでバージョンを固定します。

mise のロックファイルは `mise.lock` です[^lockfile][^mise-lock]。`mise.toml` で緩いバージョンを宣言したまま、`mise.lock` に解決済みの具体的なバージョンとチェックサムを記録し、プロジェクト内でインストールするバージョンをそろえます[^lockfile]。npm の `package-lock.json` や Cargo の `Cargo.lock` と同じ役割を果たします[^mise-lock]。ロックファイルの名前は、設定ファイルの拡張子を `.toml` から `.lock` に変えた名前です。`mise.toml` には `mise.lock` が、`mise.local.toml` には `mise.local.lock` が対応します[^lockfile]。

### ロックファイルを有効にする

ロックファイルは `lockfile` 設定で制御します[^lockfile]。`mise settings lockfile=true` を実行すると、グローバルで有効になります。プロジェクト単位の場合は、`mise.toml` の `[settings]` セクションへ書きます。次の例は、プロジェクト単位でロックファイルを有効にします。

```toml:mise.toml
[settings]
lockfile = true

[tools]
node = "22"
```

`lockfile` 設定の既定値は未設定（None）です[^lockfile]。未設定の場合、ロックファイルは有効（`true` と同じ）に扱われますが、後述する `locked` モードとは競合しません[^lockfile]。`lockfile = false` を明示すると、ロックファイルを無効にします[^lockfile]。

### ロックファイルを生成する

ロックファイルは自動では生成されません[^lockfile][^mise-lock]。`mise.lock` を作るには、空の `mise.lock` を用意してから `mise install` を実行するか、`mise lock` を実行します[^lockfile][^cli-lock]。次の例は、`mise.toml` のあるディレクトリで空の `mise.lock` を作り、`mise install` でツールを導入してロックファイルを埋めます。

```shell
$ touch mise.lock
$ mise install
```

一度 `mise.lock` を作ると、mise はツールのインストールやアップグレードのたびに `mise.lock` を更新します[^mise-lock]。生成された `mise.lock` には、ツールごとにバージョン・backend・プラットフォーム別のチェックサムとダウンロード URL が記録されます。次の例は、`jq` を宣言したプロジェクトで生成した `mise.lock` の一部です。

```toml:mise.lock
# @generated - this file is auto-generated by `mise lock`

[[tools.jq]]
version = "1.7"
backend = "aqua:jqlang/jq"

[tools.jq."platforms.macos-x64"]
checksum = "sha256:4b4568fb5c6cbb8f1a8f640ff601c6d2bbecf9fa8f2cf796c7482e7054a01c41"
url = "https://github.com/jqlang/jq/releases/download/jq-1.7/jq-macos-amd64"
```

先頭行の `# @generated` コメントには、実際には mise 公式ドキュメントの URL が続きます。本書では可読性のため URL を省略します。`mise.lock` は複数のプラットフォームのチェックサムと URL を記録します。`macos-x64`・`macos-arm64`・`linux-x64` などのプラットフォーム別のエントリを持つため、OS や CPU アーキテクチャの異なる環境でも同じバージョンを再現できます[^lockfile]。記録するプラットフォームは `lockfile_platforms` 設定で絞り込めます[^lockfile]。

`mise.lock` はバージョン管理システムにコミットして共有します[^mise-lock]。`mise.toml` と `mise.lock` の両方を共有すると、緩いバージョン指定を保ちながら、チームの全員が同じバージョンとチェックサムでツールを導入できます。一方、`mise.local.lock` は個人用の `mise.local.toml` に対応するため、`.gitignore` で除外します[^mise-lock]。

### CI での厳密な再現

`locked` 設定を有効にすると、mise はロックファイルに記録済みの URL がない場合に `mise install` を失敗させます[^settings-locked]。GitHub などへの API 呼び出しを避け、ロックファイルに固定された URL だけでインストールするため、再現性が高まります。`locked` は `mise install` の `--locked` オプションや `MISE_LOCKED=1` 環境変数でも指定できます[^cli-install][^settings-locked]。`locked` を使うには `lockfile` が有効である必要があります[^settings-locked]。CI での再現性の確保は第 16 章で扱います。

:::message
`lockfile` 設定と `locked` 設定は役割が異なります。`lockfile` はロックファイルを読み書きするかを制御します。`locked` はロックファイルに URL がないときにインストールを失敗させ、ロックファイルへの追従を強制します。チームでロックファイルを共有する場合は `lockfile` を有効にし、CI で厳密な再現を求める場合に `locked` を併せて使います。
:::

## mise trust で設定ファイルを信頼する

mise には、信頼していない設定ファイルを無条件には読み込まない仕組みがあります[^cli-trust]。`mise.toml` は環境変数の設定やコードの実行につながる記述を持てるため、mise は設定ファイルを解析する前に信頼を確認します[^cli-trust]。クローンしたばかりのリポジトリの `mise.toml` は信頼されていない状態にあり、信頼するまで mise は一部の設定を読み込みません。

### 信頼が必要になる場面

mise は、信頼していない設定ファイルに対して、対話的なシェルでは確認を促し、確認できない状況ではエラーで停止します[^cli-trust]。次の例は、`[env]` を含む信頼前の `mise.toml` があるディレクトリで `mise ls` を実行した結果です。mise は設定ファイルが信頼されていないと報告し、`mise trust` を案内します。出力は要点の行を抜粋したもので、実際には案内の行末にドキュメントの URL が続きます。

```shell
$ mise ls
mise ERROR error parsing config file: ~/work/my-app/mise.toml
mise ERROR Config files in ~/work/my-app/mise.toml are not trusted.
Trust them with `mise trust`.
```

すべての設定ファイルに信頼が必要なわけではありません。`min_version`、プレーンなバージョン文字列だけの `[tools]`、テンプレートやツールオプションを含まない `[tasks]` だけで構成された設定ファイルは、信頼の確認なしに読み込まれます[^cli-trust]。読み込み時にコードを実行する記述がないためです。`[env]` でテンプレートを使う場合や、`_.source` でスクリプトを読み込む場合（第 13 章）は、信頼が必要になります。

### 設定ファイルを信頼する

設定ファイルを信頼するには、プロジェクトのディレクトリで `mise trust` を実行します[^cli-trust]。引数なしで実行すると、現在のディレクトリの設定ファイルを信頼します。

```shell
$ mise trust
mise trusted ~/work/my-app
```

`mise trust` の主なオプションを次の表にまとめます[^cli-trust]。

| オプション | 動作 |
| --- | --- |
| `--show` | 設定ファイルの信頼状態を表示する。信頼の状態は変更しない |
| `--untrust` | 信頼を取り消す。以降は再び確認を促す |
| `--ignore` | 信頼せず、以降は設定ファイルを無視する |
| `-a`（`--all`） | 現在のディレクトリと親ディレクトリの全設定ファイルを信頼する |

信頼の状態は `mise trust --show` で確認します。次の例は、信頼済みのプロジェクトでの出力です。

```shell
$ mise trust --show
~/work/my-app: trusted
```

mise は信頼の情報を、設定ファイル本体ではなく mise の状態ディレクトリ（`~/.local/state/mise/trusted-configs`）に記録します。信頼はディレクトリのパスに紐づくため、設定ファイルをコミットしても信頼の状態は共有されません。各開発者がクローン後に自分の環境で `mise trust` を実行します。

:::message
特定のパス以下の設定ファイルを常に信頼するには、`MISE_TRUSTED_CONFIG_PATHS` 環境変数にパスを設定します[^config-env]。複数のパスは、PATH 環境変数と同じくコロン（`:`）で区切ります。信頼するパスをあらかじめ決めておく運用に使えます。設定ファイルの信頼は、コードの実行につながりうる設定を読み込む前の確認です。信頼する前に `mise.toml` の内容を確認します。
:::

## プロジェクトの mise.toml の実例とオンボーディング

本章で扱った内容をまとめて、チーム共有を想定した `mise.toml` の例と、新メンバーのオンボーディングの流れを示します。

### 共有する mise.toml の例

次の `mise.toml` は、ツールのバージョンを宣言し、ロックファイルを有効にし、環境変数とタスクを定義します。Node.js のバージョンは緩い指定にし、解決されるバージョンは `mise.lock` で固定します。

```toml:mise.toml
[settings]
lockfile = true        # mise.lock でバージョンを固定する

[tools]
node = "22"            # 22 系。解決されるバージョンは mise.lock で固定
jq = "1.7"             # JSON 処理ツール

[env]
NODE_ENV = "development"   # アプリケーションの動作環境
_.file = ".env"            # 機密情報を含む .env を読み込む

[tasks.test]
description = "テストを実行する"
run = "node --test"
```

`.gitignore` には、コミットしないファイルを並べます。`mise.toml` と `mise.lock` はコミットし、`.env` と個人用の設定は除外します。

```text:.gitignore
mise.local.toml
mise.local.lock
.env
```

### 新メンバーのオンボーディングの流れ

新たに参加した開発者は、次の手順でプロジェクトの開発環境を再現します。手順 1 は初回の一度だけ実施し、手順 2 以降をプロジェクトごとに実施します。

1. mise を導入してシェルを有効化する（第 3 章）。
2. リポジトリをクローンし、プロジェクトのディレクトリへ移動する。
3. `mise trust` で `mise.toml` を信頼する。
4. `mise install` でツールを導入する。
5. `.env` を用意し、開発を始める。

次の例は、手順 2 から手順 4 までを実行する流れです。クローンから信頼、ツールの導入までを順に行います。

```shell
$ git clone https://github.com/example/my-app.git
$ cd my-app
$ mise trust
mise trusted ~/work/my-app
$ mise install
mise node@22.20.0 ✓ installed
mise jq@1.7 ✓ installed
```

`mise install` の完了後、プロジェクトのディレクトリでは `mise.toml` が宣言したツールが PATH 上で有効になります。`mise.lock` を共有しているため、導入されるバージョンはチームの全員でそろいます。手順 5 では、共有された `.env.example` を基に `.env` を用意します。`mise run test` でタスクを実行でき、新メンバーはチームと同じ手順で作業を始められます。

## 本章のまとめ

- `mise.toml` をバージョン管理システムにコミットして共有します。`mise.toml`・`mise.lock`・ファイルタスクをコミットし、`mise.local.toml`・`.env`・`mise.local.lock` は `.gitignore` で除外します。
- 引数なしの `mise install` は、`mise.toml` の `[tools]` に宣言された全ツールを一括で導入します。クローンした開発者は一度実行するだけで環境がそろいます。
- ロックファイル `mise.lock` は、解決済みのバージョンとプラットフォーム別のチェックサムを固定します。`lockfile` 設定で有効にし、`touch mise.lock && mise install` または `mise lock` で生成します。`mise.lock` はコミットして共有します。
- `mise trust` は、信頼していない設定ファイルを mise が読み込まない仕組みに対する信頼の付与です。プレーンな `[tools]` だけの設定は信頼不要ですが、`[env]` のテンプレートやスクリプトを含む設定は信頼が必要です。信頼はパスに紐づくため、各開発者がクローン後に実行します。
- 新メンバーは、mise の導入（第 3 章）→ クローン → `mise trust` → `mise install` → 開発開始、の流れで同じ環境を再現します。CI での再現は第 16 章で扱います。

[^mise-version]: mise のバージョンは執筆時点の安定版です。最新版は mise 公式リポジトリのリリースで確認できます。<https://github.com/jdx/mise/releases>
[^secrets]: mise の機密情報管理（sops・age への対応、experimental である点）。mise 公式ドキュメント「Secrets」<https://mise.jdx.dev/environments/secrets/>
[^cli-install]: `mise install` コマンド（引数なしで `mise.toml` の全ツールを導入する点・並列インストール・インストール先・`--locked`）。mise 公式ドキュメント「mise install」<https://mise.jdx.dev/cli/install.html>
[^lockfile]: ロックファイルの設定（`lockfile` 設定・既定値 None・`mise.lock` の命名規則・生成方法・`lockfile_platforms`）。mise 公式ドキュメント「Settings」の lockfile 節 <https://mise.jdx.dev/configuration/settings.html#lockfile>
[^mise-lock]: ロックファイル `mise.lock` の概要（役割・自動生成されない点・コミットの方針・`mise.local.lock` を除外する点）。mise 公式ドキュメント「mise.lock Lockfile」<https://mise.jdx.dev/dev-tools/mise-lock.html>
