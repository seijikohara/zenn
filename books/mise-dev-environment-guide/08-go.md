本章では、Go 本体とその周辺ツールを mise で管理する方法を扱います。読了後には、Go のバージョンを `mise.toml` で固定し、Go 製の CLI ツールも同じ設定ファイルにまとめて宣言できるようになります。あわせて、Go 1.21 以降が備える toolchain 自動取得と mise の固定の関係を整理します。

本章は第 3 章で mise を導入・有効化し、第 5 章でバージョン操作を理解した状態を前提とします。mise 自体のインストールは第 3 章、`mise.toml` の構文は第 4 章、`mise install` / `mise use` / `mise ls` などの操作は第 5 章で扱いました。本章はコマンドの基本操作を再説明せず、Go 固有の事情に集中します。

本章のコマンド出力やバージョン番号は、執筆時点（2026 年 6 月、mise 2026.6.10[^mise-version]）の値です。実行する時期やツールの更新状況によって、表示されるバージョンは変わります。

## Go のバージョン管理

mise は Go を core backend で組み込み対応します[^go]。core backend は、mise が標準で内蔵するツールの取得方法です。Go は backend の追加設定なしに、ツール名 `go` で宣言できます。backend の仕組みの詳細は第 12 章で扱います。

mise は、既定では Google が配布する Go SDK のプリビルドアーカイブ（ビルド済みの実行ファイル）を取得します[^go]。取得元は `go.download_mirror` 設定で指定し、既定値は `https://dl.google.com/go` です[^go]。ソースからのコンパイルは行わないため、インストールは短時間で完了します。

### バージョンの指定方法

`go` のバージョン指定には、第 4 章で説明した構文を使います。完全指定・部分指定に加えて、`latest` のエイリアスを利用できます[^go]。

```toml:mise.toml
[tools]
go = "1.25.11"  # 完全指定。1.25.11 に固定
```

部分指定は、先頭の一部だけを書きます。`go = "1.25"` は、1.25 系の最新バージョン（執筆時点では 1.25.11）に解決されます。`latest` は最新の安定版に解決します。

```toml:mise.toml
[tools]
go = "1.25"  # 1.25 系の最新に解決
```

Go 1.20 以前のマイナーバージョンを部分指定する場合は、`prefix:` を付けます[^go]。Go 1.20 以前は各系列の初版が `.0` の接尾辞なしで公開されたため、`go = "1.20"` は初版の `1.20` への完全一致と解釈されます。`prefix:1.20` と書くと、mise は 1.20 で始まる最新版（執筆時点では 1.20.14）に解決します。

```toml:mise.toml
[tools]
go = "prefix:1.20"  # 1.20 系を部分指定
```

:::message
`latest` と部分指定は、インストール時点での最新版に解決します。インストール後はバージョンが固定され、自動では更新されません。再現性が必要なプロジェクトでは、完全指定でバージョンを固定します。
:::

### インストールと確認

`mise use` でバージョンを宣言し、インストールします。次の例は、プロジェクトのディレクトリで Go 1.25.11 を宣言します。

```shell
$ mise use go@1.25.11
mise go@1.25.11  ✓ installed
mise ~/work/my-app/mise.toml tools: go@1.25.11
```

インストール後、`go version` で有効なバージョンを確認します。

```shell
$ go version
go version go1.25.11 darwin/amd64
```

### .go-version を読み込む

mise は、Go 向けのバージョンファイルである `.go-version` を読み込めます[^go]。mise は、asdf のレガシーバージョンファイルにならって、`.go-version` を idiomatic version files と呼びます[^idiomatic]。

ただし、idiomatic version files の読み込みは既定で無効です[^idiomatic]。`.go-version` を置いただけでは、mise は Go のバージョンを解決しません。読み込みを有効にするには、`idiomatic_version_file_enable_tools` 設定にツール名を追加します。

```shell
$ mise settings add idiomatic_version_file_enable_tools go
```

設定を追加すると、mise は `.go-version` の内容を Go のバージョンとして解決します。次の例は、`1.25.11` を記述した `.go-version` を読み込んだ結果です。`Source` 列が `.go-version` を指します。

```shell
$ mise ls go
Tool  Version   Source                       Requested
go    1.25.11   ~/work/my-app/.go-version    1.25.11
```

:::message
idiomatic version files が既定で無効になったのは、mise 2025.10.0 からです[^idiomatic-default]。それ以前のバージョンでは既定で有効でした。新規のプロジェクトでは `mise.toml` を使い、`.go-version` は他ツールからの移行や互換性の維持に用います。
:::

## go backend による Go 製ツールの導入

Go 製の CLI ツールは、Go のソースコードから `go install` でインストールできます。mise は、`go install` 相当の取得を担う go backend を備えます[^go-backend]。go backend は、ツール名の接頭辞に `go:` を付け、続けて Go のモジュールパスを書く記法で宣言します[^go-backend]。

```toml:mise.toml
[tools]
go = "1.25.11"
"go:golang.org/x/tools/cmd/goimports" = "latest"  # go backend で goimports を導入
```

ツール名としての core backend の `go`（Go 本体）と、backend としての `go:`（Go 製ツールの導入）は別物です。`go` は Go の SDK をインストールします。`go:<モジュールパス>` は、指定したモジュールを `go install` でビルドし、実行ファイルを導入します。

go backend は、Go 本体が PATH 上にあることを前提とします[^go-backend]。前掲の例のように `[tools]` で `go` を宣言すると、mise が用意した Go を使ってツールをビルドできます。インストール後の確認は、ツールごとのコマンドで行います。次の例は、コードフォーマッタの goimports[^goimports] を確認します。

```shell
$ mise use go@1.25.11 "go:golang.org/x/tools/cmd/goimports@latest"
mise go@1.25.11                                    ✓ installed
mise go:golang.org/x/tools/cmd/goimports@0.46.0    ✓ installed
$ goimports -h
usage: goimports [flags] [path ...]
  -d	display diffs instead of rewriting files
  ...
```

`goimports -h` は、使い方の概要に続けてフラグの一覧を出力します。前掲の例は冒頭だけを抜粋し、フラグの一覧を `...` で省略しています。

go backend では、ビルド時のオプションも指定できます[^go-backend]。`tags` でビルドタグを、`install_env` で `go install` 実行時の環境変数を渡せます[^go-backend]。例えばデータベースドライバを選択するビルドタグは、`tags` で指定します。

```toml:mise.toml
[tools]
"go:github.com/golang-migrate/migrate/v4/cmd/migrate" = { version = "latest", tags = "postgres" }
```

:::message
backends の全体像と使い分けは第 12 章で扱います。本章では、Go 本体を扱う core backend の `go` と、Go 製ツールを導入する go backend の `go:` の区別に絞って説明します。
:::

## Go の toolchain 自動取得との関係

Go 1.21 以降、`go` コマンドは標準ライブラリ・コンパイラなどを含む toolchain を同梱します[^go-toolchain]。`go` コマンドは、同梱の toolchain に加えて、PATH 上で見つかる別バージョンや、必要に応じてダウンロードした別バージョンの toolchain も使えます[^go-toolchain]。どの toolchain を使うかは、環境変数 `GOTOOLCHAIN` の値と、`go.mod`（または `go.work`）の `go` 行・`toolchain` 行で決まります[^go-toolchain]。

`go.mod` の `go` 行は、モジュールが要求する Go の最小バージョンを宣言します[^go-toolchain]。`toolchain` 行は、推奨する toolchain のバージョンを宣言します[^go-toolchain]。

```go:go.mod
module example.com/my-app

go 1.26.0
```

`GOTOOLCHAIN` の既定値は `auto` です[^go-toolchain]。`auto` の場合、同梱の toolchain が `go` 行・`toolchain` 行の要求と同等以上に新しければ、同梱の toolchain を使います[^go-toolchain]。要求がより新しいときは、`go` コマンドは PATH 上で該当バージョンを探し、見つからなければダウンロードして使います[^go-toolchain]。

ここで、mise が固定した Go と食い違いが起きえます。mise が `go = "1.25.11"` を固定し、`go.mod` の `go` 行が `1.26.0` を要求する場合を考えます。`GOTOOLCHAIN=auto` のもとで、`go` コマンドは Go 1.26 系の toolchain をダウンロードして実行します。mise の `mise.toml` は Go 1.25.11 を指すまま、実際にビルドに使われる toolchain は Go 1.26 系になります。

食い違いを避けるには、`GOTOOLCHAIN` を `local` に設定します[^go-toolchain]。`GOTOOLCHAIN=local` の場合、`go` コマンドは常に同梱の toolchain（mise が固定した Go）を使い、自動取得と切り替えを行いません[^go-toolchain]。`go.mod` の `go` 行が同梱より新しいバージョンを要求すると、`go` コマンドはエラーを報告して終了します[^go-toolchain]。mise で固定したバージョンを唯一の toolchain とする運用では、`GOTOOLCHAIN=local` を設定します。mise の `[env]` セクションで、プロジェクト単位に環境変数を設定できます。

```toml:mise.toml
[tools]
go = "1.25.11"

[env]
GOTOOLCHAIN = "local"  # 自動取得を無効化し、mise が固定した Go を使う
```

:::message
`GOTOOLCHAIN=local` を設定すると、`go.mod` が要求する Go のバージョンと mise が固定したバージョンの食い違いがエラーとして表面化します。エラーが出た場合は、`mise.toml` の `go` のバージョンを引き上げるか、`go.mod` の `go` 行を下げて、両者をそろえます。`[env]` セクションによる環境変数の管理は第 13 章で扱います。
:::

## プロジェクトの mise.toml の実例

本章で扱った設定をまとめて、Go プロジェクトの `mise.toml` の例を示します。Go のバージョンを固定し、`GOTOOLCHAIN=local` で toolchain を固定したうえで、Go 製ツールの goimports を go backend で宣言します。

```toml:mise.toml
[tools]
go = "1.25.11"                                       # Go のバージョンを固定
"go:golang.org/x/tools/cmd/goimports" = "latest"     # コードフォーマッタを go backend で導入

[env]
GOTOOLCHAIN = "local"  # mise が固定した Go を toolchain として使う
```

設定後、`mise install` でツールをまとめてインストールします。インストール後、Go 本体とツールのバージョンを確認します。

```shell
$ mise install
mise go@1.25.11                                    ✓ installed
mise go:golang.org/x/tools/cmd/goimports@0.46.0    ✓ installed
$ go version
go version go1.25.11 darwin/amd64
$ go env GOTOOLCHAIN
local
```

go backend のツールは、`mise install` の出力でモジュールパスと解決後のバージョンを表示します。`latest` で宣言した goimports は、執筆時点では 0.46.0 に解決します。`go version` の出力の末尾 `darwin/amd64` は、実行環境の OS とアーキテクチャを表します。Apple Silicon の Mac では `darwin/arm64`、Linux では `linux/amd64` のように、実行環境に応じて変わります。

`mise.toml` をバージョン管理システムにコミットすると、チームの全員が同じバージョンの Go と goimports を使い、同じ toolchain でビルドできます。

## 本章のまとめ

- mise は Go を core backend で組み込み対応します。ツール名 `go` で宣言し、完全指定・部分指定・`latest` でバージョンを指定します。1.20 以前のマイナーバージョンの部分指定には `prefix:` を付けます。
- mise は既定で Google が配布する Go SDK のプリビルドアーカイブを取得します。取得元は `go.download_mirror` 設定で指定します。
- mise は `.go-version` を読み込めますが、mise 2025.10.0 以降は既定で無効です。`idiomatic_version_file_enable_tools` にツール名を追加して有効にします。
- Go 製の CLI ツールは go backend（`go:<モジュールパス>`）で導入します。core backend の `go`（Go 本体）と go backend の `go:`（Go 製ツールの導入）は別物です。
- Go 1.21 以降は `GOTOOLCHAIN=auto` のもとで toolchain を自動取得するため、mise が固定した Go と食い違う場合があります。`GOTOOLCHAIN=local` を設定すると、mise が固定した Go を唯一の toolchain として使います。

[^mise-version]: mise のバージョンは執筆時点の安定版です。最新版は mise 公式リポジトリのリリースで確認できます。<https://github.com/jdx/mise/releases>
[^go]: mise での Go の管理（core backend・Go SDK のプリビルドアーカイブの取得・`go.download_mirror` 設定・バージョン指定・`prefix:`・`.go-version` の読み込み）。mise 公式ドキュメント「Go」<https://mise.jdx.dev/lang/go.html>
[^idiomatic]: mise が読み込む idiomatic version files（`.go-version` など）と、既定で無効である点、`idiomatic_version_file_enable_tools` 設定。mise 公式ドキュメント「Configuration」<https://mise.jdx.dev/configuration.html>
[^idiomatic-default]: idiomatic version files の既定が無効に変わった経緯（mise 2025.10.0）。mise 公式リポジトリの Discussion「Disabling idiomatic version files by default」<https://github.com/jdx/mise/discussions/4345>
[^go-backend]: go backend の記法（`go:<モジュールパス>`・`go install` 相当・Go 本体が PATH 上にある前提・`tags`・`install_env` オプション）。mise 公式ドキュメント「Go Backend」<https://mise.jdx.dev/dev-tools/backends/go.html>
[^goimports]: goimports は import 文を整理し、gofmt 相当の整形をする公式ツールです。不足する import を追加し、未使用の import を削除します。Go 公式「golang.org/x/tools/cmd/goimports」<https://pkg.go.dev/golang.org/x/tools/cmd/goimports>
[^go-toolchain]: Go の toolchain 選択（Go 1.21 以降の同梱 toolchain・`GOTOOLCHAIN` の値・`auto` と `local` の挙動・`go.mod` の `go` 行と `toolchain` 行）。Go 公式ドキュメント「Go Toolchains」<https://go.dev/doc/toolchain>
